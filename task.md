# Особенности архитектуры
## Максимальная инкапсуляция и модульность
Каждый логический элекмент проекта должен быть максимально изолирован от других и легко заменён на аналогичный. 

Пример 1
Если сейчас основной интерфейс - это телеграм-бот, то завтра это может быть веб-приложением на streamlit, а послезавтра - что-то совершенно новое. 
А для отладки можно вообще консоль использовать. 
Или всё это использовать одновременно и переключать через параметр

Пример 2
Базовая опция логирования - postgres. Но завтра может потребовать перейти на slq-lite (для компактности) или clickhouse (для масштабирования). 
Нужно хранить все параметры и изолировать функции так, чтобы не завязываться на особенности функционала конкретных баз данных или даже отколючить логирование целиком.

Пример 3
Для работы могут использовать сторонние API. Эти провайдеры тоже могут меняться. Процесс получения и обработки данных оттуда и сама внутреняя логика должны быть надёжно разнесены

## Предельная простота
Я хочу понимать код проекта, по крайней мере на верхнем уровне. 
Не нужно использовать излишне сложные синтаксические конструкции. В идеале - попытаться воспроизвести мой стиль написания кода.
Если какая-то функция может быть из-за повышенных требований к прототе и читабельности может стать более длинной, менее изящной - это нормально. 
Но производительность страдать не должна.  

## Документируем всё
Пишем максимально подробные комментарии на русском языке. У каждого компонента пишем его цель, назначение, основную мысль и то, что получается на выходе.
Если нужно выполнить какие-то необычные вещи, то указываем это в комментарии явно.
Если тут есть сложность, опасность  или риски необычного поведения в будущем - указываем это явно в комментариях.
Если есть что-то, что можно будет улучшить в будущем, но сейчас нет в промте, то пишем TODO комментарии (и отдельно выносим их в будущий таск-лист).
Полное описание сигнатур функций и других элементов по стандартно PEP8

## Гибкое отношение к зависимостям
Над проектом не будет постоянной поддержки, поэтому улучшения, проверки и дополнения - всегда проблема. 
Если есть возможность взять внешний функционал, который проверен, который постоянно развивается, разработчики которого могут учесть больше мелочей - нужно использовать.
Например, для создания ботов в телеграм лушче использовать aiogram, а не просто API telegram. 

Но, разумеется, не имеет смысла запаковывать в проект большую внешнюю библиотеку типа sklearm ради одногой простой функции типа max()

## Производительность и асинхронность
Пытаемся находить оптимальное решение по памяти и нагрузке процессеора среди возможных. Но с сохраненияем условий выше. 
Не обязательно найти иделаьный оптимальный алгоритм. Важно не остановиться на откровенно плохом и неудачном. 

Сразу держим в фокусе асинхронность вызовов. Ожидаем, что продуктом одновременно будут пользоваться тысячи пользователей. 

## Избыточное тестирование
Тестирование очень важно. 
Делаем два слоя тестирования.

Первый слой - это стандартные проверки и авто-тесты, которые являются составной частью кода. Прохождение этих тестов обязательно для коммита. 
Второй слой - это бизнес-логика. Это набор тестов, условий и проверок, которые я буду задавать отдельно.
Возможно такое, что в часть тестов в этих слоях будет проверять одно и то же и даже добуквенно совпадать. Это нормально. 


## Разделение на prod и stage
Ветка main - это prod, то, что видят пользователи. Предполагается, что он работает на внешней виртуальной машине.
Ветка stage - это stage, среда для тестирования. Предполагается, что работает локально на компьютере тестировщика.  

# Важные компоненты
## Внутреняя логика
"Мозг" всей системы. В ней на основе полученной информации генерируются смыслы. 
Может быть достаточно сложной. 

## Коллектор информации
Собирает данные из различных источников, приводит их к виду, с которым уже может работать "мозг".

## Интерфейс работы с пользователем
Телеграм-бот, веб-приложение на streamlit или просто консоль
Как получаем информацию, так и выводим её 

## Внутренее логирование
Пишем важные лони во втутренюю базу. 
Типовые таблицы:
- events, фикисруем взаимодействие с интрфейсом (клики по кнопкам, появление окон, вывод результатов): timestamp, user_id, event_name, chanel, event_parameters, inserted_at  
- users, фиксируем статусы пользователя: user_id, user_name, registration_date, registration_chanel, last_active_at, is_paid, is_trial, ia_activate
- таблицы со специфичной логикой
Таблицы в stage средне имеют те же названия, просто развернуты на другой базе (с локальным ip)
Есть опция выключения ботов

# MVP, с которого начинаем
Приложение, спршивает как вас зовут, после этого отправляет в главное меню. 
В главном меню 3 пункта: 
- узнать текущую температуру в Москве
- прочитать случайный анекдот
- дорогой дневник: написать в свободной форме мысли о текущем дне

user_id присваиваем инкерментально
имя пользователя сохраняем в user_id
время регистрации - время ввода user_name

Температуру узнаём по API
Случайный анекдот - случайный параметр из json

Логирование в postgres, выключается по параметру в yaml.
Вывод по умолчанию в streamlit, может измениться на telegram или консоль по параметру в yaml.

event_name для логирования: 
- start_screen_visit
- main_menu_visit
- option1_visit
- option2_visit
- option3_visit
- registration
- diary_message_sent


Тестирование логики:
- все кнопки кликабельны
- все нужные действия логируются
- параметры в users перезаписываются
- нет коллизий по user_id
- специальные символы не ломают отправку сообщений и запись в логах
- лаг в телеграм меньше 8 секунл
- лаг в стримлит меньше 8 секунд
